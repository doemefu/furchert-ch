## 2. plan

**Goal:** `/automation` renders pixel-faithfully to `pages.jsx` and CTAs
deep-link to a real `/automation/scan`; `/automation/scan` renders a clear
4-step ETHON-styled mockup with a visible "Demo / Mockup" banner and zero
backend / outbound traffic; build/lint/typecheck pass; sitemap includes
`/automation` but excludes `/automation/scan`.

### Context
**Summary:** Two routes. The landing is a verbatim port + one FAQ
accordion client island. The scanner is a single client wizard with mock
in-memory state, no `fetch`, no server action, no `/api/scan/*` route; the
"Analyse erstellen" button advances to a static "demo report" view that is
explicitly labelled as illustrative (not a real analysis). All content
authored DE+EN.

### Options considered

**Option 1 — Verbatim landing port + ETHON wizard with explicit demo banner (chosen)**
- What: `/automation/page.tsx` server component verbatim from prototype +
  `FAQAccordion.tsx` client island. `/automation/scan/page.tsx` server
  page (metadata + container + banner) wrapping `ScanWizard.tsx` client.
  Bilingual data in `src/data/{automation-faqs,scan}.ts`; short labels
  in next-intl `pages.automation` + `pages.scan`.
- Pros: Pixel-faithful where the prototype exists; ETHON-faithful where it
  doesn't; zero backend; clearly-labelled mockup honours CLAUDE.md non-
  negotiables; minimum client JS.
- Cons: Authoring real DE+EN copy for the entire spec §2.8 flow is bulky.

**Option 2 — Single client page for both routes (rejected)**
- What: One `'use client'` tree managing landing + wizard with internal nav.
- Rejected: kills SSG/SEO for the landing; mixes a server-renderable page
  with interactive state for no benefit.

**Option 3 — Server-rendered wizard with form-submission steps (rejected)**
- What: Wizard as multi-route (`/scan/step-1`, `/step-2`, …) with form POST
  between steps.
- Rejected: Would require server actions / `/api/scan/*`, contradicting the
  non-negotiable "no scan API, no persistence". Even ephemeral cookie state
  edges into "scan backend" territory. Keep state purely in-browser.

**Option 4 — Skip `/scan` entirely until Phase 7 (rejected)**
- What: `/automation` only; hero #scan CTA becomes a contact deep-link.
- Rejected: Master plan Phase 3 explicitly includes a scan mockup; this
  preserves the visual product story while honouring the mockup rule.
